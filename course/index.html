How ccmux Works
Module 1

Your AI's Money-Saving Bodyguard

What ccmux does and why it exists

The Problem

You use Claude — an AI assistant — through an API. Every request costs money, charged per token. Anthropic offers three models at wildly different price tiers, but your tooling sends everything to the same one.

"What's 2+2?" doesn't need a $15-per-million-token brain. But a complex multi-file refactor absolutely does.

🧠

Opus — $15/M tokens

The heavyweight. Complex reasoning, multi-file refactors, architectural decisions.

⚙️

Sonnet — $3/M tokens

The workhorse. Day-to-day coding, writing tests, explaining code.

⚡

Haiku — $0.80/M tokens

The speedster. Quick lookups, simple Q&A, formatting tasks.

💸
The Cost Gap

Opus is 19x more expensive than Haiku. If even half your requests are simple, you're burning money on every one.

The Solution

ccmux is a local proxy that sits between your CLI and Anthropic's API. It reads each request, estimates how complex it is, and picks the cheapest model that can handle it.

Think of it like an airport security checkpoint. It scans your luggage (reads the request metadata) without opening it (no body modification), then routes you to the right lane — economy or first class — based on what the scanner sees.

1
You launch Claude normally

ccmux starts a local HTTP server and spawns your Claude CLI as a child process.

2
Intercept

Your CLI sends its API call to ccmux instead of Anthropic, because ccmux swapped the destination address (more on this in Screen 3).

3
Analyze

ccmux reads the request headers and lightweight signals — system prompt length, conversation turns, tool count — to gauge complexity.

4
Pick a model

Based on a rule set you control, ccmux selects Haiku, Sonnet, or Opus. Simple question? Haiku. Gnarly refactor? Opus.

5
Forward

ccmux acts as a reverse proxy: it forwards the original request — byte-for-byte — to Anthropic with the chosen model, and streams the response back via SSE.

The Env Var Trick

How does ccmux redirect traffic without you changing a single line of code? It uses an environment variable. When ccmux spawns your CLI, it overrides ANTHROPIC_BASE_URL to point at itself.

src/lifecycle/wrapper.ts 
export function buildChildEnv(
  parentEnv: NodeJS.ProcessEnv,
  port: number,
  token: string,
): NodeJS.ProcessEnv {
  const baseUrl = `http://127.0.0.1:${port}`;
  const noProxy = '127.0.0.1,localhost';
  return {
    ...parentEnv,
    ANTHROPIC_BASE_URL: baseUrl,
    NO_PROXY: noProxy,
    no_proxy: noProxy,
    NOPROXY: noProxy,
    CCMUX_PROXY_TOKEN: token,
  };
}
PLAIN ENGLISH

Define a function that builds the set of environment variables for the child process (your Claude CLI).

It takes your current environment, the port ccmux is listening on, and an internal auth token.

Build the address of our local proxy: http://127.0.0.1:<port> — this is just "localhost" with the specific port number.

Tell the OS to bypass any corporate or system-wide proxy for local traffic, so requests actually reach ccmux.

Copy all existing env vars with the spread operator, then override ANTHROPIC_BASE_URL to point to our proxy — this is the magic trick that redirects every API call without touching the client code.

Set three variants of the no-proxy variable (NO_PROXY, no_proxy, NOPROXY) because different Node.js HTTP libraries check different casing.

Pass an internal token so ccmux can verify the request actually came from the child process it spawned, not some random local connection.

💡
The Interception Pattern

This technique — changing where software sends its requests by swapping an environment variable — is used everywhere in software. Proxies, testing tools, and service meshes all work this way. It is one of the most powerful tricks in a developer's toolkit.

What ccmux Does NOT Do

ccmux is intentionally minimal. It makes three hard guarantees so you can trust it with production API traffic.

Zero telemetry No outbound calls except to api.anthropic.com. Your data never leaves the expected path.
No body modification Request and response bodies pass through byte-for-byte. ccmux only reads headers and lightweight signals.
Auth passthrough Your Anthropic API key is forwarded as-is. ccmux never stores, logs, or inspects it.

Meet the Cast

The five actors that make routing decisions

The Five Actors

Think of ccmux as a film production crew. Every request that arrives is a new scene to shoot, and each crew member has exactly one job. Nobody steps on anyone else's role.

🎬
Proxy Server (Fastify)

The director -- receives every request, coordinates the pipeline, forwards the result. This is the reverse proxy that Claude Code talks to instead of Anthropic directly.

🔎
Signal Extractor

The script supervisor -- reads every incoming request and extracts signals like token count, tool usage, frustration markers, and plan-mode flags. Takes notes on every take so the rest of the crew can make informed decisions.

📋
Policy Engine

The casting director -- evaluates YAML rules in order using first-match-wins logic. When a rule matches the extracted signals, it picks the model. No drama, no debate.

🎯
Classifier

The talent scout -- makes judgment calls when no rule matches. Uses a heuristic scoring algorithm and, as a fallback, a quick Haiku-based classification.

📝
Decision Logger

The editor -- records every routing decision to a daily JSONL log file with structured JSON via pino. Everything goes on the record for later review.

Where They Live

Each actor gets its own folder. One folder, one responsibility, zero overlap.

src/
proxy/ The director -- HTTP server, routing handler, health check
signals/ Script supervisor -- extracts plan mode, tokens (via tiktoken), tools, frustration
policy/ Casting director -- YAML rules, first-match-wins evaluator, recipes
classifier/ Talent scout -- heuristic scorer + Haiku-based backup classifier
logging/ Editor -- decision logs, privacy redaction, structured JSON
cli/ Stage manager -- parses command-line args, launches the server
config/ Props department -- loads YAML config, validates schemas, watches for changes via chokidar for hot-reload
lifecycle/ Safety crew -- graceful shutdown, session cleanup, signal handling
dashboard/ Screening room -- real-time stats, cost tracking, decision history
types/ Wardrobe -- shared TypeScript interfaces and type definitions

The Decision Pipeline

How ccmux decides which model gets your request

Every request that reaches ccmux passes through a sorting facility before it ships. Each package (request) gets scanned, checked against routing rules, weighed by a classifier if the rules have no opinion, and sent down the right conveyor belt to Haiku, Sonnet, or Opus.

The Five Stages

Every POST /v1/messages request passes through these five stages before reaching Anthropic. The entire pipeline runs synchronously and adds less than 5ms of latency.

📦
The Sorting Facility Metaphor

Think of this like a package sorting facility. Stage 1 reads the label. Stage 2 scans the contents. Stage 3 checks automated routing rules. Stage 4 asks a human (the classifier) if no rule covers it. Stage 5 puts it on the right truck.

What Gets Scanned

In stage 2, the proxy extracts a set of signals from each request. These are the data points the routing engine uses to make its decision.

planMode Is the AI in plan mode? Detected from system prompt or headers.
estInputTokens Approximate input size, estimated by tiktoken.
toolUseCount How many tool use invocations the AI has made so far.
messageCount How deep into the conversation we are — more turns often means more complex context.
frustration Frustration detection — catches phrases like “that is wrong” or “try again.”
retryCount How many times this exact request has been retried.
fileRefCount Number of file paths mentioned in the conversation — more files usually means a bigger task.
💡
Graceful Degradation

Each signal extractor is wrapped in try/catch. If one fails (say, token counting hits a weird edge case), that one signal gets a default value — the rest still work fine. One broken sensor does not crash the whole system.

Reading the Scanner Output

This is the actual return value from the signal extractor. Each safe() wrapper means “try this, but if it crashes, use the fallback value instead.” The result is frozen so nothing downstream can accidentally mutate it.

src/signals/extract.ts:134-149 
return Object.freeze({
  planMode: safe(logger, 'plan-mode', null, () =>
    detectPlanMode(body?.system)),
  messageCount: safe(logger, 'messageCount', 0, () =>
    messages.length),
  tools: toolNames,
  toolUseCount: safe(logger, 'tool-use-count', 0, () =>
    countToolUse(messages)),
  estInputTokens: safe(logger, 'tokens', 0, () =>
    estimateInputTokens(body?.system, messages)),
  fileRefCount: safe(logger, 'file-refs', 0, () =>
    countFileRefs(messages)),
  retryCount: safe(logger, 'retry', 0, () =>
    retryCount(hash, sessionContext)),
  frustration: safe(logger, 'frustration', null, () =>
    detectFrustration(messages)),
  explicitModel: safe(logger, 'explicit-model', null, () =>
    explicitModelOf(body)),
});
PLAIN ENGLISH

Lock this object so nothing can change it after creation. Immutability prevents downstream code from accidentally corrupting the signals.

planMode: Check if the AI is in planning mode by reading the system prompt. Falls back to null (unknown) if detection fails.

messageCount: Count how many messages are in the conversation. Falls back to 0 if the array is malformed.

tools: List all the tool names available in this request. No safe() needed — already extracted above.

toolUseCount: Scan content blocks to count how many times the AI has actually invoked tools.

estInputTokens: Estimate token count using tiktoken. This avoids a round-trip to the API just to measure size.

fileRefCount: Regex-scan messages for file paths. More files referenced usually correlates with higher complexity.

retryCount: Hash the request and check the session context to see if this exact request has been sent before.

frustration: Look for correction patterns in user messages. Falls back to null rather than 0 because “no data” differs from “no frustration.”

explicitModel: Check if the user explicitly requested a specific model (via HMAC-signed header or body field). If set, it overrides all other routing.

⏱
TTL and Signal Freshness

Signals like retryCount and frustration rely on session state that has a TTL. If the user walks away for 30 minutes, the session expires and those signals reset to defaults. This prevents stale frustration from a morning session from escalating an afternoon request to Opus.

Module 4

The Smart Parts

Rules, scoring, and how ccmux actually picks a model

The Rule Book

Think of ccmux as a wine sommelier. It examines the occasion (request signals), consults the wine list (your YAML policy rules), and picks the right bottle. Here is the actual rule book from the balanced recipe.

src/policy/recipes/balanced.yaml 
rules:
  # Plan mode deserves Opus.
  - id: plan-to-opus
    when: { planMode: true }
    then: { choice: opus }

  # Trivial turns: short, no tools, small context.
  - id: trivial-to-haiku
    when:
      all:
        - { messageCount: { lt: 5 } }
        - { toolUseCount: { eq: 0 } }
        - { estInputTokens: { lt: 2000 } }
    then: { choice: haiku }

  # Retries suggest the first tier couldn't cope.
  - id: retry-escalate
    when: { retryCount: { gte: 2 } }
    then: { escalate: 1 }

  # Frustration signal -> bump a tier.
  - id: frustration-escalate
    when: { frustration: true }
    then: { escalate: 1 }
PLAIN ENGLISH

Here are the routing rules, checked top to bottom...

Rule 1: If the AI is in planning mode, always use Opus (the best model).

Rule 2: If ALL of these are true — short conversation, no tools used, small input — use Haiku (cheapest).

Rule 3: If this request has been retried 2+ times, bump up one tier (the current model is struggling).

Rule 4: If the user sounds frustrated, bump up one tier.

🍷
The Sommelier Analogy

Rules are the wine list — explicit pairings the sommelier memorized. "Steak? Cabernet." "Fish? Sauvignon Blanc." Written in a DSL anyone on your team can read and edit.

How Rules Compose

Each rule's when clause uses composition to combine simple conditions into complex ones. Three building blocks cover every case.

ALL
Every condition must be true

Like a checklist — all boxes must be checked. Used for the "trivial" rule: short AND no tools AND small input.

ANY
At least one condition must be true

Like a multiple-choice question — any correct answer works.

NOT
The condition must be false

Like a bouncer with an exclusion list — "anyone EXCEPT VIPs."

⚠️
First Match Wins

Rules are checked top-to-bottom. The FIRST rule that matches wins — later rules are never evaluated. If NO rule matches, the engine says "abstain" and the heuristic classifier takes over.

The Scoring Machine

When no rule matches, the sommelier switches from the wine list to their trained palate — the heuristic scorer. It examines five signals and assigns points to produce a confidence score.

📏

Token Count

Under 500 → 0 pts. 500-2k → 1 pt. 2k-8k → 2 pts. Over 8k → 3 pts.

🔧

Tool Breadth

0.5 pts per unique tool, capped at 3 pts. More tools = more complex task.

📝

Code Fences

0.3 pts per code block, capped at 2 pts. Code in the conversation = technical work.

📂

File References

0.4 pts per file path, capped at 2 pts. File paths suggest multi-file work.

💬

Phrasing

Imperative verbs like "write" or "build" → +1 pt. Questions → -1 pt.

The points are summed and mapped to a band — like wine price tiers on the list.

0-3
Haiku

Low complexity. Simple questions, short context, no tools. The house wine — reliable and affordable.

3-6.5
Sonnet

Medium complexity. Some tools, moderate context, code blocks. A solid mid-range bottle.

6.5+
Opus

High complexity. Many tools, large context, imperative commands. Break out the reserve vintage.

The Backup Expert

The heuristic scorer is deterministic — fast and predictable, but it can only count signals, not understand meaning. When enabled, ccmux also asks a "senior sommelier": it sends a lightweight classification request to Haiku itself, racing against an 800ms timeout.

If Haiku responds in time and its confidence score is higher than the heuristic's, Haiku's recommendation wins. If Haiku is slow or uncertain, the heuristic result stands. Best of both worlds.

💡
Sticky Sessions Prevent Whiplash

Once ccmux picks a model for your session, it caches that choice (keyed by HMAC of session signals) with a TTL of 2 hours. This is called a sticky session. It prevents jarring mid-conversation model switches where you suddenly go from Opus-quality answers to Haiku-quality answers.

Under the Hood

Here is the actual scoring function. Each signal contributes points, they are summed, clamped to a 0–10 range, and converted to a token band.

src/classifier/heuristic.ts:119-140 
const tokenContribution = tokenBand(signals.estInputTokens);
const toolContribution = Math.min(
  signals.tools.length * WEIGHTS.toolBreadthFactor,
  WEIGHTS.toolBreadthCap,
);
const codeContribution = Math.min(
  countCodeFences(input.body) * WEIGHTS.codeBlockFactor,
  WEIGHTS.codeBlockCap,
);
const phrasingContribution = phrasingDelta(lastUserText(input.body));
const fileContribution = Math.min(
  signals.fileRefCount * WEIGHTS.filePathFactor,
  WEIGHTS.filePathCap,
);

const rawScore =
  tokenContribution + toolContribution + codeContribution +
  phrasingContribution + fileContribution;
const score = clamp(rawScore, 0, 10);
const suggestedModel = bandFromScore(score);
PLAIN ENGLISH

Look up the token band (0–3 points based on input size).

Count unique tools × 0.5, but cap at 3 points.

Count code blocks × 0.3, cap at 2 points.

Check if the last message starts with an imperative verb (+1) or is a question (-1).

Count file references × 0.4, cap at 2 points.

Add up all the scores.

Clamp to 0–10 range so no single outlier dominates.

Convert score to model: under 3 = Haiku, 3–6.5 = Sonnet, over 6.5 = Opus.

🍷
Trained Palate, Not a Crystal Ball

The heuristic scorer is like a sommelier's instinct — it uses observable signals (what it can see on the label) to make a fast, good-enough call. It doesn't read the full message, just like a sommelier doesn't taste every bottle before recommending it. When the escalate rules or the Haiku classifier override it, that is the senior sommelier stepping in.

Pick Your Style

When you run ccmux init, you choose one of three recipes. Think of them as driving modes: eco, sport, and comfort. Each generates a complete YAML config you can customize later.

⚖️

Balanced (default)

Opus for planning, Haiku for trivial turns, escalate on retries and frustration. Good starting point for most users.

💰

Aggressive

Maximum cost savings. Haiku by default, Sonnet for tool use, Opus only for plan mode. Best if you are optimizing for cost.

🎯

Conservative

Maximum quality. Opus by default, Sonnet only for simple questions. Best for critical work where accuracy matters most.

⚖️
Start Balanced, Then Tune

The balanced recipe is the safest on-ramp. Run it for a week, review ccmux report, and then tighten or loosen rules based on what the data tells you. The ccmux tune command will even suggest specific changes.

Watching the Gauges

The ccmux dashboard command launches a local SPA built with React that shows routing activity in real time. The ccmux report command gives you the same data as a terminal summary.

💡
Shadow Mode: Test Before You Commit

Set mode to "shadow" and ccmux will log what it WOULD have done without actually changing the model. This lets you preview cost savings and catch bad rules before they affect your work.

Every routing decision is recorded to a daily JSONL log. Here is what each entry contains:

timestamp When the decision was made
signals All extracted signals (tokens, tools, plan mode, etc.)
matchedRule Which rule fired (or "classifier" if no rule matched)
chosenModel The final model selection (haiku, sonnet, or opus)
confidence How confident the system was in its choice (0.2 to 0.85)
costEstimate Estimated cost of this request at the chosen tier
🔄
Hot Reload

Edit your config file while ccmux is running -- it watches for changes and picks them up automatically. No restart needed. Logs rotate daily with automatic log rotation so they never grow unbounded.
