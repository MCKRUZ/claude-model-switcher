Module 4

The Smart Parts

Rules, scoring, and how ccmux actually picks a model

The Rule Book

Think of ccmux as a wine sommelier. It examines the occasion (request signals), consults the wine list (your YAML policy rules), and picks the right bottle. Here is the actual rule book from the balanced recipe.

src/policy/recipes/balanced.yaml 
rules:
  # Plan mode deserves Opus.
  - id: plan-to-opus
    when: { planMode: true }
    then: { choice: opus }

  # Trivial turns: short, no tools, small context.
  - id: trivial-to-haiku
    when:
      all:
        - { messageCount: { lt: 5 } }
        - { toolUseCount: { eq: 0 } }
        - { estInputTokens: { lt: 2000 } }
    then: { choice: haiku }

  # Retries suggest the first tier couldn't cope.
  - id: retry-escalate
    when: { retryCount: { gte: 2 } }
    then: { escalate: 1 }

  # Frustration signal -> bump a tier.
  - id: frustration-escalate
    when: { frustration: true }
    then: { escalate: 1 }
PLAIN ENGLISH

Here are the routing rules, checked top to bottom...

Rule 1: If the AI is in planning mode, always use Opus (the best model).

Rule 2: If ALL of these are true — short conversation, no tools used, small input — use Haiku (cheapest).

Rule 3: If this request has been retried 2+ times, bump up one tier (the current model is struggling).

Rule 4: If the user sounds frustrated, bump up one tier.

🍷
The Sommelier Analogy

Rules are the wine list — explicit pairings the sommelier memorized. "Steak? Cabernet." "Fish? Sauvignon Blanc." Written in a DSL anyone on your team can read and edit.

How Rules Compose

Each rule's when clause uses composition to combine simple conditions into complex ones. Three building blocks cover every case.

ALL
Every condition must be true

Like a checklist — all boxes must be checked. Used for the "trivial" rule: short AND no tools AND small input.

ANY
At least one condition must be true

Like a multiple-choice question — any correct answer works.

NOT
The condition must be false

Like a bouncer with an exclusion list — "anyone EXCEPT VIPs."

⚠️
First Match Wins

Rules are checked top-to-bottom. The FIRST rule that matches wins — later rules are never evaluated. If NO rule matches, the engine says "abstain" and the heuristic classifier takes over.

The Scoring Machine

When no rule matches, the sommelier switches from the wine list to their trained palate — the heuristic scorer. It examines five signals and assigns points to produce a confidence score.

📏

Token Count

Under 500 → 0 pts. 500-2k → 1 pt. 2k-8k → 2 pts. Over 8k → 3 pts.

🔧

Tool Breadth

0.5 pts per unique tool, capped at 3 pts. More tools = more complex task.

📝

Code Fences

0.3 pts per code block, capped at 2 pts. Code in the conversation = technical work.

📂

File References

0.4 pts per file path, capped at 2 pts. File paths suggest multi-file work.

💬

Phrasing

Imperative verbs like "write" or "build" → +1 pt. Questions → -1 pt.

The points are summed and mapped to a band — like wine price tiers on the list.

0-3
Haiku

Low complexity. Simple questions, short context, no tools. The house wine — reliable and affordable.

3-6.5
Sonnet

Medium complexity. Some tools, moderate context, code blocks. A solid mid-range bottle.

6.5+
Opus

High complexity. Many tools, large context, imperative commands. Break out the reserve vintage.

The Backup Expert

The heuristic scorer is deterministic — fast and predictable, but it can only count signals, not understand meaning. When enabled, ccmux also asks a "senior sommelier": it sends a lightweight classification request to Haiku itself, racing against an 800ms timeout.

If Haiku responds in time and its confidence score is higher than the heuristic's, Haiku's recommendation wins. If Haiku is slow or uncertain, the heuristic result stands. Best of both worlds.

💡
Sticky Sessions Prevent Whiplash

Once ccmux picks a model for your session, it caches that choice (keyed by HMAC of session signals) with a TTL of 2 hours. This is called a sticky session. It prevents jarring mid-conversation model switches where you suddenly go from Opus-quality answers to Haiku-quality answers.

Under the Hood

Here is the actual scoring function. Each signal contributes points, they are summed, clamped to a 0–10 range, and converted to a token band.

src/classifier/heuristic.ts:119-140 
const tokenContribution = tokenBand(signals.estInputTokens);
const toolContribution = Math.min(
  signals.tools.length * WEIGHTS.toolBreadthFactor,
  WEIGHTS.toolBreadthCap,
);
const codeContribution = Math.min(
  countCodeFences(input.body) * WEIGHTS.codeBlockFactor,
  WEIGHTS.codeBlockCap,
);
const phrasingContribution = phrasingDelta(lastUserText(input.body));
const fileContribution = Math.min(
  signals.fileRefCount * WEIGHTS.filePathFactor,
  WEIGHTS.filePathCap,
);

const rawScore =
  tokenContribution + toolContribution + codeContribution +
  phrasingContribution + fileContribution;
const score = clamp(rawScore, 0, 10);
const suggestedModel = bandFromScore(score);
PLAIN ENGLISH

Look up the token band (0–3 points based on input size).

Count unique tools × 0.5, but cap at 3 points.

Count code blocks × 0.3, cap at 2 points.

Check if the last message starts with an imperative verb (+1) or is a question (-1).

Count file references × 0.4, cap at 2 points.

Add up all the scores.

Clamp to 0–10 range so no single outlier dominates.

Convert score to model: under 3 = Haiku, 3–6.5 = Sonnet, over 6.5 = Opus.

🍷
Trained Palate, Not a Crystal Ball

The heuristic scorer is like a sommelier's instinct — it uses observable signals (what it can see on the label) to make a fast, good-enough call. It doesn't read the full message, just like a sommelier doesn't taste every bottle before recommending it. When the escalate rules or the Haiku classifier override it, that is the senior sommelier stepping in.
