Meet the Cast

The five actors that make routing decisions

The Five Actors

Think of ccmux as a film production crew. Every request that arrives is a new scene to shoot, and each crew member has exactly one job. Nobody steps on anyone else's role.

🎬
Proxy Server (Fastify)

The director -- receives every request, coordinates the pipeline, forwards the result. This is the reverse proxy that Claude Code talks to instead of Anthropic directly.

🔎
Signal Extractor

The script supervisor -- reads every incoming request and extracts signals like token count, tool usage, frustration markers, and plan-mode flags. Takes notes on every take so the rest of the crew can make informed decisions.

📋
Policy Engine

The casting director -- evaluates YAML rules in order using first-match-wins logic. When a rule matches the extracted signals, it picks the model. No drama, no debate.

🎯
Classifier

The talent scout -- makes judgment calls when no rule matches. Uses a heuristic scoring algorithm and, as a fallback, a quick Haiku-based classification.

📝
Decision Logger

The editor -- records every routing decision to a daily JSONL log file with structured JSON via pino. Everything goes on the record for later review.

Where They Live

Each actor gets its own folder. One folder, one responsibility, zero overlap.

src/
proxy/ The director -- HTTP server, routing handler, health check
signals/ Script supervisor -- extracts plan mode, tokens (via tiktoken), tools, frustration
policy/ Casting director -- YAML rules, first-match-wins evaluator, recipes
classifier/ Talent scout -- heuristic scorer + Haiku-based backup classifier
logging/ Editor -- decision logs, privacy redaction, structured JSON
cli/ Stage manager -- parses command-line args, launches the server
config/ Props department -- loads YAML config, validates schemas, watches for changes via chokidar for hot-reload
lifecycle/ Safety crew -- graceful shutdown, session cleanup, signal handling
dashboard/ Screening room -- real-time stats, cost tracking, decision history
types/ Wardrobe -- shared TypeScript interfaces and type definitions

Notice the pattern

The Proxy Server is the only actor that talks to the outside world. Signals, Policy, and Logger never see the raw HTTP request -- they only receive the data the Proxy hands them. The Classifier was not even needed this time because Policy found a match.

The Director's Setup

Here is how the Proxy Server registers its three routes. This is the entry point where every request first lands.

src/proxy/server.ts function registerRoutes(app: FastifyInstance, opts: ProxyServerOptions): void { const hot = makeHotPathHandler({ logger: opts.logger }); const through = passThrough({ logger: opts.logger }); const health = makeHealthHandler({ startTimeMs: Date.now(), version: '0.0.0', mode: 'passthrough', }); app.post('/v1/messages', hot); app.route({ method: ['GET', 'POST', 'PUT', 'PATCH', 'DELETE', 'OPTIONS', 'HEAD'], url: '/v1/*', handler: through }); app.get('/healthz', health); }
Plain English
Define a function that wires up all the server's URL routes.
Create the smart routing handler for the main AI endpoint.
Create a pass-through handler that forwards everything else unchanged.
Create a health check endpoint that reports uptime, version, and current mode.
 
 
 
 
When a POST hits /v1/messages -- the main AI endpoint -- use the smart router.
All other API paths under /v1/ get forwarded to Anthropic unchanged.
The /healthz endpoint lets monitoring tools check if ccmux is alive.
 
💡
Only one route is "smart"

The entire signal extraction, policy evaluation, and classification pipeline only runs for POST /v1/messages. Every other endpoint passes straight through -- ccmux does not interfere with model listings, token counting, or any other API call. The HMAC-signed decision log only records the smart-routed calls.

Separation of Concerns

Each actor does exactly one thing well. The proxy does not know how to score complexity. The classifier does not know how to forward HTTP requests. This pattern -- giving each piece one clear job -- is one of the most important ideas in software engineering.
